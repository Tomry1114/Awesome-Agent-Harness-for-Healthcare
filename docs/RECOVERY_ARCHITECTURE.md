# Bounded Clinical Recovery (BCR) — Unified Recovery Architecture (v3)

Design v3. Supersedes v2 ("Committed Workflow Completion" as the whole-system name). v3 demotes CWC to ONE of
four recovery paths, separates four layers so no benchmark/workflow knowledge hides in a substrate adapter, and
fixes the metric/safety/binding confusions surfaced in review. No benchmark names in kernel/substrate; grounded in
the real repo `/hpc2hdd/home/ce483/Medical_harness/`. Design only — not production code.

## 0. What v3 corrects vs v2

- **CWC is a sub-module, not the system.** v2 called MedCTA's answer revision a CWC "commit" — wrong: A→B is
  newly-generated medical content, not the completion of an already-decided plan. The system is **Bounded Clinical
  Recovery**; CWC is one branch.
- **MedCTA = Evidence Acquisition + Agent Re-entry + Non-regression Acceptance.** The harness ACQUIRES missing
  evidence and returns control to the ROOT AGENT to re-reason; B is generated by the agent, never by a
  harness judge/comparator. AnswerRetention only *accepts/rejects* B.
- **Four-layer separation.** Workflow knowledge (what an order/appeal/prior-auth needs) is a **Workflow Module**,
  NOT part of the substrate adapter. A GUI adapter never knows "appeal fields"; a FHIR adapter never knows "order".
- **No commitment ≠ FAILED.** Distinct declined/not-applicable states + a proper metrics taxonomy.
- **≤ 1 irreversible commit per episode by default.** Multi-commit only under a declared transaction contract.
- **`system_metadata` binding source** for operational fields (never clinical payload).
- Split `unique_workflow_structure` → `ArgumentBinding` / `AffordanceBinding` / `ProtocolConstant`;
  `reversible_write` → `staged_write` (reversibility declared by manifest); add `BLOCKED_UNRESOLVED_AFFORDANCE`;
  `resolve_commitments() → list`; confidence≥0.8 is an experimental acceptance policy, not a correctness claim.
- **Preserved from v2 (the core innovation):** the **Information-Sufficiency Boundary** — the harness may complete
  the deterministic execution closure of a committed goal and stops only at a new decision/content/unknown
  parameter. Also preserved: oracle-blind; Evidence Ledger; the auth machine; exact auth-id; ActionExecutor;
  before/after_action; bounded execution; controlled-replay attribution; root-agent provenance; no-repeat-after-UNKNOWN.

## 1. The four recovery paths (Gap Router)

```
Gap Router (classify the failure)
├── Evidence gap      → Evidence Acquisition → root-agent RE-ENTRY → Non-regression Acceptance
├── Execution gap     → Committed Workflow Completion (CWC): deterministic bounded plan
├── Verification gap  → Idempotent Reconciliation (re-read only; never re-commit)
└── Decision gap      → BLOCKED_NEEDS_DECISION (the harness must not decide)
```

Only the **execution** path performs authorized environment mutation via CWC. The **evidence** path performs
read-only acquisition and then hands the agent back the wheel (no harness-authored answer). The **verification**
path only re-reads. The **decision** path always blocks. A single benchmark can use different paths for different
tasks; a single episode uses exactly one path.

## 2. Four-layer separation (the anti-specialization rule)

```
1. Recovery Kernel      state machine · gap routing · authorization · budget · provenance ·
                        UNKNOWN reconciliation · per-step execution · metrics.
                        Knows: paths, steps, bindings, auth tiers. Knows NOT: FHIR/GUI/image, order/appeal.

2. Substrate Adapter    HOW to observe/act in this environment: FHIR read/create/update; GUI snapshot/resolve/
                        click/type/upload; perceptual ImageDescription/RegionAttribute/OCR; state read-back.
                        Knows NOT: order/appeal/prior-auth/MedCTA — no workflow, no benchmark fields.

3. Workflow Module      WHAT a clinical/admin process is: CreateOrderWorkflow, AppealSubmissionWorkflow,
   (Clinical Process    PriorAuthorizationWorkflow, EvidenceAcquisitionWorkflow. Declares steps, required
    Pack)               bindings, affordance targets (by role/label), postconditions, transaction contract.
                        Substrate-agnostic in intent; calls the substrate adapter for primitives.

4. Benchmark Adapter    Normalizes PB/HAB/MedCTA task fields; maps lifecycle events (deliverable_confirmed /
                        before_final) and environment state-paths. Does NOT change kernel or workflow rules.
```

**Consequence for the interface (P0-2 fix).** v2's `RecoveryAdapter.compile_workflow` is split:

```
SubstrateAdapter:                         # environment mechanics only
    resolve_affordance(target_spec, observation) -> AffordanceBinding | BLOCKED_*
    execute_primitive(kind, action, auth) -> Outcome        # read/navigate/staged_write/irreversible_commit
    read_state(paths) -> state_view                          # authoritative read-back
    classify_result(result) -> EvidenceState

WorkflowModule:                           # process knowledge; substrate-agnostic
    match_goal(committed_goal, ctx) -> bool                  # does this workflow realize this goal?
    required_bindings(goal, ctx) -> [BindingSpec]            # semantic args this workflow needs
    compile_plan(goal, ctx) -> Plan(steps=[WorkflowStep(kind, affordance_target, arg_specs)])
    verify_effect(goal, state_view) -> bool | None

BenchmarkAdapter:                         # task/field/lifecycle/state-path normalization
    context(task) -> ctx
    resolve_commitments(root, trajectory, goal, judge, ctx) -> [CommittedGoal]   # oracle-blind
    should_trigger(lifecycle_event) -> bool
    state_path(logical_name) -> concrete state path          # e.g. "appeal_submitted" -> payer_a_state...submittedAppeal
```

The Kernel wires them: BenchmarkAdapter resolves commitments → WorkflowModule.match_goal selects the process →
WorkflowModule.compile_plan produces the plan → Kernel walks it, resolving each step's affordance via
SubstrateAdapter.resolve_affordance and each arg via the binding system, executing via
SubstrateAdapter.execute_primitive under tiered auth.

## 3. Recovery Kernel — states, one-commit rule, metrics

### 3.1 States

Progress (execution path): `NOT_STARTED → ACQUIRING → PLANNING → EXECUTING_STAGED → READY_TO_COMMIT → COMMITTING
→ VERIFYING → VERIFIED`. Evidence path: `NOT_STARTED → ACQUIRING → AGENT_REENTRY → ACCEPTED / KEPT_ORIGINAL`.
Terminals: `BLOCKED_NEEDS_DECISION`, `BLOCKED_MISSING_EVIDENCE`, `BLOCKED_AMBIGUOUS_TARGET`,
`BLOCKED_UNRESOLVED_AFFORDANCE`, `FAILED`, `UNKNOWN`, `ALREADY_REALIZED`.
Declined (not failures): `NOT_APPLICABLE` (no workflow matches), `DECLINED_NO_COMMITMENT` (no committed goal).

### 3.2 One-irreversible-commit rule (P0-4)

Default invariant: **a Recovery Episode contains AT MOST ONE `irreversible_commit` step.** After it, only
`verify` / read-back / `staged_write` documentation / reconcile steps may run. A WorkflowModule may declare
`transaction_contract = {commits:[...], compensation:[...], partial_landing_policy:...}` to permit >1 irreversible
commit; absent that contract, a plan with >1 irreversible commit is a compile error → the goal must be split into
multiple `CommittedGoal`s / episodes. This bounds partial-completion risk to a single durable effect per episode
(the existing no-repeat-after-UNKNOWN + read-back verify already covers the single-commit case).

### 3.3 One goal per episode (次-4)

`resolve_commitments() → list[CommittedGoal]`. The Kernel runs **one episode per goal**, each with its own dedup
key, authorization, execution, verification, and attribution. Multiple committed orders in one deliverable = N
independent episodes, not one plan.

### 3.4 Metrics taxonomy (P0-3)

Per task, count: **eligible** (a gap-classified recoverable episode was possible) · **engaged** (a plan/acquire
actually ran) · **verified_recovery** (VERIFIED / ACCEPTED) · **correctly_blocked** (a BLOCKED_* / DECLINED_* /
NOT_APPLICABLE terminal — a *correct* refusal, reported as its own bucket) · **failed_recovery** (FAILED) ·
**unknown_recovery** (UNKNOWN). Outcome effect is reported as F→T, T→F, unchanged, plus average extra tool cost.
`DECLINED_NO_COMMITMENT` / `NOT_APPLICABLE` / `BLOCKED_*` are NEVER counted as failures.

## 4. Binding system (P0-5, 次-1)

Three distinct binding kinds — do not conflate:

```
ArgumentBinding(name, value, source, provenance)      # WHERE a parameter VALUE comes from
AffordanceBinding(target_spec, ref, observation_hash) # WHERE the environment control is (a located ref)
ProtocolConstant(name, value)                         # a schema-FIXED value (status="active", intent="order")
```

`ArgumentBinding.source` legal sources, split into two classes:

- **Semantic parameters** (change clinical/business meaning: drug, dose, diagnosis, rationale, disposition,
  claim, patient, order_type). Sources: `agent_commitment` | `authoritative_state` | `bound_evidence`. An
  untraceable semantic arg ⇒ BLOCK. **`system_metadata` is FORBIDDEN for semantic params.**
- **Operational metadata** (do NOT constitute a decision: timestamp/authoredOn, request_id, idempotency_key,
  recovery_tag, episode_id, client-generated resource id). Source: **`system_metadata`** — permitted ONLY for
  fields a substrate/workflow schema explicitly whitelists as operational. Never for clinical payload.

`unique_workflow_structure` from v2 is REMOVED as a value source: "there is exactly one Submit button" is an
`AffordanceBinding` (control location), and "the form fixes Dispute Type = Appeal" is a `ProtocolConstant` — neither
proves a *clinical value* is correct. This closes the loophole where "the only field" was treated as content proof.

## 5. Step kinds (次-2, 次-3)

`RecoveryStep.kind ∈ {read, navigate, staged_write, irreversible_commit, verify, acquire}`. `staged_write`
(renamed from `reversible_write`) does NOT assume reversibility; the substrate manifest declares per-tool
`{side_effect_scope, rollback_available, autosave_possible, server_persisted}`, and the Kernel uses those to pick
the auth tier and the read-back check — never the action name. A `staged_write` whose manifest says
`server_persisted:true` is authorized like an irreversible tier.

Tiered authorization: read/navigate/acquire → read gate, no auth minted (strict reader). staged_write → scoped
authorization sized by the manifest's declared side-effect scope. irreversible_commit → exact single-use
`MutationAuthorization(tier=irreversible)` + idempotency_key, no re-commit after UNKNOWN. `MutationAuthorization`
gains one `tier` field; the AVAILABLE→…→CANCELLED machine + exact_scope_match are unchanged.

Affordance failure has its own terminal: a target that cannot be uniquely located in the live observation →
`BLOCKED_UNRESOLVED_AFFORDANCE` (0 candidates or an ordering violation) or `BLOCKED_AMBIGUOUS_TARGET` (>1
candidate, no disambiguating bound id) — distinct from `BLOCKED_MISSING_EVIDENCE`.

## 6. The four paths in detail

### 6a. Evidence Acquisition + Agent Re-entry + Non-regression Acceptance  (MedCTA; also FHIR/GUI evidence gaps)

The harness does NOT author the revised answer. Flow: detect an evidence gap (a claim/decision unsupported by
acquired observation) → **ACQUIRE** the missing evidence via read-only `acquire` steps (perceptual read whose
region arg is derived ONLY from the public question; FHIR/GUI read) → **return control to the ROOT AGENT** with the
new evidence appended to its context (agent re-entry / re-execution) → the agent produces B itself → **Non-regression
Acceptance** (`evaluate_candidate`/AnswerRetention) accepts B over A only if the delta is non-regressive and
evidence-supported, else keep A. B's authorship = agent; the harness only acquires + gates. This is NOT a CWC
commit and mints no MutationAuthorization. Report replay F→T/T→F/unchanged/coverage-gain/extra-tool-cost; the
acceptance threshold (e.g. judge conf) is an experimental policy, not a calibrated correctness guarantee (次-5).

### 6b. Committed Workflow Completion (CWC)  (PB orders; HAB submit_auth / decision-documentation)

`CommittedGoal → WorkflowModule.compile_plan → per-step [DecisionBoundary(resolve semantic ArgumentBindings) +
AffordanceBinding(locate control) + tiered execute] → ≤1 irreversible_commit → Effect Verification`. Stops at the
first unbound SEMANTIC argument. Operational fields bind via `system_metadata`. Example workflow modules:
`CreateOrderWorkflow` (FHIR: read-govern → existing-effect probe → create → read-back), `AppealSubmissionWorkflow`
(GUI: navigate → search → open form → fill semantic fields → attach bound doc → one submit → read-back),
`PriorAuthorizationWorkflow` (GUI submit_auth: fill structured fields from authoritative EMR state → one submit).

### 6c. Idempotent Reconciliation  (verification gap)

A commit dispatched but read-back ambiguous → re-READ the same target (by id / EffectCompletionKey), never
re-commit. Resolves UNKNOWN → VERIFIED/FAILED without a second mutation.

### 6d. Decision Boundary

Any semantic argument with no legal source, any net-new content (rationale/dose/diagnosis), any ambiguous target →
the matching BLOCKED_* terminal. Reported as `correctly_blocked`, a first-class correct outcome.

## 7. Substrate × Workflow × Benchmark mapping

| dataset | path(s) | Workflow Module(s) | Substrate Adapter | recoverable population (honest) |
|---|---|---|---|---|
| PB | CWC (+ evidence prereq) | CreateOrderWorkflow | FHIR (search/create/read-back) | REAL, proven (cp3 pelvic-US); text-committed order not executed |
| HAB | CWC; some Decision-Boundary | PriorAuthorizationWorkflow, AppealSubmissionWorkflow, DecisionDocumentationWorkflow | GUI (snapshot/resolve/click/type/upload); +read payer_a/b_state | concentrated in submit_auth structured forms + decision-documentation; free-text appeals under weak agents → correctly BLOCK |
| MedCTA | Evidence Acquisition + Agent Re-entry | EvidenceAcquisitionWorkflow | perceptual (ImageDescription/Region/OCR) | REAL (answered-without-looking); harness acquires, AGENT re-answers |

Substrate fixes required before HAB: `GuiEnvReal.read_state` must also read `payer_a_state`/`payer_b_state`;
the real GUI DOM-ref `resolve_affordance` (live-observation, label/role, unique) must be built (`affordance.py` is
perception-only today).

## 8. Metrics (report per dataset)

eligible · engaged · verified_recovery · correctly_blocked · failed_recovery · unknown_recovery; outcome F→T /
T→F / unchanged; average extra tool calls per engaged episode; BLOCK-reason histogram. Attribution via controlled
replay (record agent trace → replay OFF/ON, only harness differs). T→F must be reported and driven to zero.

## 9. Implementation order

1. **Kernel v3 primitives** (behavior-preserving): 4-layer interfaces (SubstrateAdapter / WorkflowModule /
   BenchmarkAdapter), binding system (ArgumentBinding+source classes incl system_metadata / AffordanceBinding /
   ProtocolConstant), states (incl NOT_APPLICABLE/DECLINED/BLOCKED_UNRESOLVED_AFFORDANCE), one-commit rule,
   metrics taxonomy, `MutationAuthorization.tier`. Regression gate: PB cp3 still F→T via CreateOrderWorkflow +
   FhirSubstrateAdapter + PbBenchmarkAdapter (behavior-identical to today).
2. **PB path**: CreateOrderWorkflow (FHIR) — the proven case re-expressed in the 4-layer form; extends to
   medication/lab/referral; decision gaps blocked at commitment resolution.
3. **HAB substrate prereqs** (payer_state read + real resolve_affordance), then PriorAuthorizationWorkflow +
   DecisionDocumentationWorkflow (the real recoverable population); AppealSubmissionWorkflow with honest BLOCK
   boundaries.
4. **MedCTA path**: EvidenceAcquisitionWorkflow + agent re-entry + Non-regression Acceptance (no mutation).
5. **Metrics + controlled replay** per dataset; report the full taxonomy, correctly_blocked as its own bucket.
